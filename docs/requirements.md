# Требования к проекту

## Наименование компонентов в shared

Название компонентов в слое `shared` должно формироваться на основе пути к файлу компонента.

### Правило формирования имени

Имя компонента формируется путем объединения названий папок в пути (начиная с `shared`) с использованием PascalCase, где каждая папка становится частью имени.

### Примеры

- Компонент `shared/ui/components/buttons/back.tsx` должен называться `ButtonsBack`
- Компонент `shared/ui/components/forms/input.tsx` должен называться `FormsInput`
- Компонент `shared/ui/mui/button/custom.tsx` должен называться `MuiButtonCustom`

### Структура имени

```
[Путь от shared до файла в PascalCase] = Имя компонента
```

Все папки в пути (кроме `shared`) преобразуются в PascalCase и объединяются без разделителей.

## Тестирование функций в shared/lib

Для каждой функции, расположенной в `shared/lib`, должен быть написан тест.

### Требования к тестам

- Тесты должны находиться в папке `_tests/shared/lib/`
- Имя тестового файла должно соответствовать имени тестируемого файла с добавлением суффикса `.test.ts` или `.test.tsx`
- Тесты должны покрывать основную функциональность функции, включая граничные случаи и обработку ошибок

### Пример структуры

```
shared/
  lib/
    example.ts

_tests/
  shared/
    lib/
      example.test.ts
```

### Пример теста

Если функция находится в `shared/lib/example.ts`, то тест должен быть в `_tests/shared/lib/example.test.ts`.

## JSDoc комментарии для функций

Все функции должны иметь JSDoc комментарии в формате:

```typescript
/**
 * Описание функции
 *
 * @param props Пропсы
 * @param props.theme Тема
 */
```

## Кроссслойные импорты

Все кроссслойные импорты должны использовать алиас `@/`:

- `@/pages/auth` вместо относительных путей
- `@/entities/task` вместо относительных путей
- `@/features/auth` вместо относительных путей
- `@/shared/lib` вместо относительных путей
